<Callout type="analogy" title="🏠 Analogy">
A house organized "by material" — all the metal in one room, all the wood in another — is useless; you organize by <em>function</em>: a kitchen, a bathroom, a bedroom, each with the pipes and wiring it needs. Go code is the same. Folders named <code>handlers/</code>, <code>models/</code>, <code>services/</code> scatter a single feature across the house; folders named <code>order/</code>, <code>billing/</code> keep each feature whole. This page is about that architectural choice — <em>how to slice</em> — not the language mechanics.
</Callout>

## The one rule that's architectural: slice by feature, not by layer

Given the mechanics, the design decision that matters is **what your top-level packages represent**. Two repos can use the exact same `cmd/` + `internal/` conventions and end up with opposite architectures:

<Mermaid code={String.raw`graph TD
  subgraph By["❌ by technical layer"]
    H["handlers/"] --- M["models/"] --- S["services/"] --- R["repos/"]
    note1["one 'order' feature is smeared<br/>across all four folders"]
  end
  subgraph Feat["✅ by feature / domain"]
    O["order/ — types + rules + repo iface"]
    B["billing/"]
    P["platform/ — db, http adapters"]
    note2["each package is a bounded context<br/>and a future service boundary"]
  end`} />

**Layer-sliced** (`handlers/`, `models/`, `services/`) feels tidy but scatters every feature across the whole tree, produces wide cross-package dependencies, and gives you no clean line to cut when you later want to extract a service. **Feature-sliced** (`order/`, `billing/`) keeps each [bounded context](/architecture/domain-driven-design/) cohesive in one package that owns its types, its rules, **and** the repository interface it needs.

## Where the domain↔infrastructure seam goes on disk

The layout *is* your [hexagonal architecture](/architecture/hexagonal-and-clean-architecture/), expressed as directories:

```text
myapp/
├── cmd/server/main.go       # composition root: wire & start (thin)
└── internal/
    ├── order/               # feature pkg = bounded context
    │   ├── order.go         #   domain types + rules
    │   └── repository.go    #   the repo INTERFACE (what the domain needs)
    ├── billing/
    └── platform/
        ├── postgres/        # adapters: IMPLEMENT the interfaces
        └── httpapi/         #   (import the driver here, never in order/)
```

The interface lives with the domain (`order/repository.go`); the implementation lives in infrastructure (`platform/postgres`). So the import arrow points **inward** — infrastructure depends on the domain, never the reverse. (The mechanics of why `internal/` makes this a hard wall, and why cycles can't form, are in [Packages & modules](/fundamentals/packages-and-modules/).)

<Callout type="gotcha" title="⚠️ Layout is a boundary decision — get the slicing right early">
The cheap-to-change-now, expensive-to-change-later part of layout isn't <code>cmd/</code> vs <code>pkg/</code> — it's <em>by-feature vs by-layer</em>. A feature package that owns its data and exposes an interface is already a candidate <a href="/architecture/modular-monolith-vs-microservices/">service boundary</a>; a layer-sliced tree has no clean line to cut and turns a future extraction into a rewrite. So slice by domain from the start, even in a tiny app — it costs nothing then and is painful to retrofit. (Everything else about growing the tree — start flat, add <code>cmd/</code>/<code>internal/</code> under pressure — is in <a href="/fundamentals/packages-and-modules/">Packages &amp; modules</a>.)
</Callout>
