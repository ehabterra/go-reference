<Callout type="analogy" title="🛟 Analogy">
A lifeboat holds twelve. With thirty people in the water, the humane-sounding instinct is to let everyone climb in — and then the boat swamps and <em>everyone</em> drowns. Load shedding is the hard, correct call: take twelve, keep them alive, and signal for more boats. Accepting load you can't carry doesn't help the extra people; it sinks the ones you could have saved. A service past capacity is that lifeboat — shedding excess is how you keep anyone afloat.
</Callout>

## Three related tools

- **Rate limiting** — a *policy*: each client gets N requests/sec, enforced always (the [token bucket](/concurrency/rate-limiting/)). Ensures fairness and protects you from any one client.
- **Load shedding** — *self-preservation*: when the server is at capacity *right now*, reject excess (fast 429/503) based on live signals (in-flight count, latency, queue depth).
- **Backpressure** — a *signal upstream*: tell producers to slow down rather than buffering unbounded work.

## See it: load shedding with an admission limit

This runs here — a semaphore caps concurrent work; requests over the limit are **shed** immediately instead of piling up. Output is deterministic:

<Playground client:visible title="shed.go" code={String.raw`package main

import (
	"fmt"
	"sync"
	"sync/atomic"
)

func main() {
	const capacity = 5 // max concurrent requests we can handle well
	sem := make(chan struct{}, capacity)

	var served, shed int64
	var attempted sync.WaitGroup
	attempted.Add(12)
	release := make(chan struct{}) // holds admitted requests "in flight"

	// 12 requests arrive at once; we can only handle 5 concurrently.
	for i := 0; i < 12; i++ {
		go func() {
			select {
			case sem <- struct{}{}: // admitted — hold the slot while "working"
				atomic.AddInt64(&served, 1)
				attempted.Done()
				<-release // simulated in-flight work keeps the slot taken
				<-sem
			default: // at capacity — shed immediately, don't queue
				atomic.AddInt64(&shed, 1)
				attempted.Done()
			}
		}()
	}
	attempted.Wait() // every request has made its admit/shed decision
	close(release)   // let the admitted ones finish
	fmt.Printf("served: %d, shed (fast 429): %d\n", served, shed)
	fmt.Println("-> accepted work stays within capacity and succeeds;")
	fmt.Println("   excess is rejected fast instead of collapsing the service.")
}`} />

Excess requests get a fast rejection (a 429/503 to the client) rather than queueing until everything times out. The accepted ones stay within capacity and succeed — **partial availability beats total collapse**.

## Shed early, degrade gracefully

Two principles make shedding effective:

- **Reject early.** Shed at admission (the edge), *before* a request consumes a DB connection and downstream calls — otherwise you waste the very resources you're protecting. Make the rejection itself cheap.
- **Degrade gracefully.** Don't fail everything equally; shed *non-critical* work first. Serve a stale [cache](/cloud/caching-with-redis/) instead of recomputing, skip recommendations to keep checkout alive, drop low-priority background jobs before user-facing ones. The user gets a reduced-but-working service.

In Go, a **bounded channel is natural backpressure** — a full jobs channel blocks the producer, throttling the whole [pipeline](/patterns/pipeline/) instead of buffering unboundedly.

<Callout type="go" title="🐹 Combine per-client limits with global shedding">
The robust setup uses both axes: <strong>rate-limit per client</strong> (a token bucket keyed by API key/IP) so no single caller is unfair, <em>and</em> <strong>shed globally</strong> when total load exceeds capacity so a legitimate surge across many clients can't sink you. Express capacity with a bounded <a href="/concurrency/rate-limiting/">semaphore/worker pool</a>, set a hard cap on in-flight requests, return <code>429</code> with <code>Retry-After</code> so well-behaved clients back off, and classify requests by priority so degradation sheds the least-important work first. See <a href="/concurrency/rate-limiting/">rate limiting</a> for the token-bucket and semaphore mechanics.
</Callout>

<Callout type="gotcha" title="⚠️ Unbounded queues turn overload into collapse">
The instinct to 'just queue the extra requests' is how overload becomes catastrophe. An unbounded queue under sustained overload grows without limit — latency climbs until every queued request has already timed out by the time it's served (you do all the work for nothing), memory fills, and the process dies. This is congestion collapse. <strong>Bound every queue</strong>, and when it's full, shed (reject) rather than accept — a fast failure the client can retry beats a slow failure that takes the whole service down. A queue is a shock absorber, not infinite storage; past its bound, the only safe answer is 'no'.
</Callout>
