<Callout type="analogy" title="🗂️ Analogy">
A text log is a pile of handwritten notes; a structured log is a spreadsheet. With notes, finding "every failed payment for user 42 last Tuesday" means reading the whole pile. With a spreadsheet, it's a filter. In production — dozens of replicas emitting thousands of lines a second into one aggregator — you live and die by that filter. slog turns your notes into spreadsheet rows for free.
</Callout>

## Structured, leveled, queryable

In a cluster, logs from every replica stream into one aggregator (Loki, Elasticsearch, Datadog). To be useful they must be **structured** — key/value fields a machine can index — not prose you grep. Go's standard `log/slog` does this with no dependency.

## See it: slog with JSON, levels, and request-scoped fields

A JSON logger, a level, and a sub-logger carrying a `request_id` on every line. This runs here (the timestamp is dropped so the output is deterministic — in production you keep it):

<Playground client:visible title="slog.go" code={String.raw`package main

import (
	"log/slog"
	"os"
)

func main() {
	// JSON handler → stdout. ReplaceAttr drops "time" for reproducible output.
	h := slog.NewJSONHandler(os.Stdout, &slog.HandlerOptions{
		Level: slog.LevelInfo,
		ReplaceAttr: func(groups []string, a slog.Attr) slog.Attr {
			if a.Key == slog.TimeKey {
				return slog.Attr{} // omit time in the demo
			}
			return a
		},
	})
	logger := slog.New(h)

	// A request-scoped logger: every line it emits carries these fields.
	reqLog := logger.With("request_id", "req-abc123", "user", 42)

	reqLog.Info("handling request", "method", "POST", "path", "/orders")
	reqLog.Warn("slow dependency", "dep", "payments", "latency_ms", 820)
	reqLog.Error("request failed", "status", 500, "err", "payment declined")

	logger.Debug("not shown — below the Info level") // filtered out
}`} />

Each line is JSON with `level`, `msg`, `request_id`, `user`, and event-specific fields — exactly what an aggregator indexes. The `Debug` line is filtered by the level. `With()` builds the request-scoped logger so you never repeat the `request_id`.

## Correlation IDs across the request

The one field that makes logs powerful is a **correlation/request ID**: generate it at the edge (or read an incoming `X-Request-ID`), stash it in the `context`, and attach it to every log line. Then the aggregator can show one request's entire journey — across handlers, goroutines, and downstream services (propagate the ID in a header). This is the logging half of what [distributed tracing](/cloud/distributed-tracing/) formalizes.

```go
// Middleware: derive a request-scoped logger and put it in the context.
func withLogger(base *slog.Logger, next http.Handler) http.Handler {
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		id := r.Header.Get("X-Request-ID")
		if id == "" { id = newID() }
		l := base.With("request_id", id, "method", r.Method, "path", r.URL.Path)
		ctx := context.WithValue(r.Context(), loggerKey{}, l)
		next.ServeHTTP(w, r.WithContext(ctx))
	})
}
```

<Callout type="go" title="🐹 Log to stdout, let the platform route it">
Don't open log files or manage rotation in a containerized app — write the event stream to <strong>stdout/stderr</strong> and let Kubernetes + a collector (Fluent Bit, Vector) ship it. App-managed files vanish with the Pod and couple you to a filesystem you don't own. Set the <strong>level via config</strong> (env var) so you can turn on debug logging without a redeploy, use a single JSON handler app-wide, and pass the request-scoped logger through the <code>context</code> rather than a global. See <a href="/stdlib/structured-logging/">slog in the stdlib track</a> for the full API.
</Callout>

<Callout type="gotcha" title="⚠️ Logs are not metrics — and they can leak secrets">
Two traps. First, <strong>don't count with logs</strong>: parsing log lines to compute request rates or error percentages is slow and expensive — that's what <a href="/cloud/metrics-with-prometheus/">metrics</a> are for. Logs are for individual events you investigate; metrics are for aggregates you alert on. Second, logs are <strong>retained and widely readable</strong>, so never log secrets or sensitive PII — tokens, passwords, full <code>Authorization</code> headers, card numbers. Redact with a <code>slog.LogValuer</code> type, and never dump a raw request or config that contains them.
</Callout>
