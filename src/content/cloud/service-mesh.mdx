<Callout type="analogy" title="🏢 Analogy">
Imagine every employee in a company having their own personal assistant who sits beside them, handles all their calls — verifying who's on the line (mTLS), redialing if it drops (retries), hanging up if it rings too long (timeouts), logging every call (observability), and routing some calls to the new office (traffic splitting). ## What a mesh moves out of your code

In a [microservices](/cloud/microservices-basics/) system, every service needs the same cross-cutting plumbing for talking to others: **retries, timeouts, load balancing, mutual TLS, traffic routing, and telemetry**. A **service mesh** (Istio, Linkerd) provides these as an infrastructure layer so the application doesn't re-implement them.

Most meshes use the **sidecar** pattern: a proxy (often Envoy) runs beside each service instance and transparently intercepts all its traffic. A central **control plane** configures every sidecar, so you change a timeout or a canary split *centrally* without touching or redeploying the apps.

## See it: the sidecar idea, in miniature

A sidecar is conceptually a wrapper that adds retries/timeouts around a call the app makes "plainly." This in-process model shows the shape — the app calls `Get`, unaware the proxy is applying policy. This runs here:

<Playground client:visible title="sidecar.go" code={String.raw`package main

import (
	"errors"
	"fmt"
)

// The app makes a "plain" call through this interface, unaware of policy.
type Caller interface{ Get() (string, error) }

// app is the real service logic — no resilience code in it.
type app struct{ attempt int }

func (a *app) Get() (string, error) {
	a.attempt++
	if a.attempt < 3 { // flaky upstream for the first 2 tries
		return "", errors.New("connection reset")
	}
	return "200 OK from service B", nil
}

// sidecar WRAPS the caller and adds mesh policy (retries) transparently.
type sidecar struct {
	inner      Caller
	maxRetries int
}

func (s sidecar) Get() (string, error) {
	var err error
	for i := 0; i <= s.maxRetries; i++ {
		var resp string
		if resp, err = s.inner.Get(); err == nil {
			fmt.Printf("[sidecar] success on attempt %d\n", i+1)
			return resp, nil
		}
		fmt.Printf("[sidecar] retry %d after: %v\n", i+1, err)
	}
	return "", err
}

func main() {
	// The mesh injects the sidecar; app code is unchanged.
	var svc Caller = sidecar{inner: &app{}, maxRetries: 3}
	resp, err := svc.Get()
	fmt.Println("app sees:", resp, err)
}`} />

The application has *zero* resilience code — the sidecar supplies it. A real mesh does this at the network layer (so it works for any language) and adds mTLS, routing, and telemetry the same transparent way.

<Callout type="go" title="🐹 Go can do resilience in-code — decide where the line is">
Go's stdlib already gives you the building blocks a mesh provides per-call: <code>context</code> deadlines for <a href="/cloud/resilience-patterns/">timeouts</a>, retry-with-backoff loops, <code>crypto/tls</code> for encryption, and connection pooling in <code>http.Transport</code>. For a <strong>few</strong> Go services, doing this in a shared internal library is often simpler and lower-overhead than running a mesh. The mesh wins when you have <strong>many</strong> services (especially polyglot), want to change policy centrally without redeploys, or need fleet-wide mTLS and uniform telemetry. Rule of thumb: start with in-code resilience (<a href="/cloud/grpc-and-service-comms/">gRPC interceptors</a> are a great place for it), and adopt a mesh when fleet scale or zero-trust makes per-service maintenance the bigger cost.
</Callout>

<Callout type="gotcha" title="⚠️ A mesh is a distributed system you now also operate">
The sidecar model isn't free: an extra proxy beside every workload adds <strong>latency per hop</strong> and <strong>CPU/memory overhead</strong>, and the control plane is one more complex system to run, upgrade, and debug (now your outage might be in the mesh, not your code). For small systems this complexity usually outweighs the payoff — don't adopt a mesh for three services. Also: the mesh handles <em>transport</em> resilience, not <em>business</em> correctness — it can retry a request, but making retries safe still requires <a href="/cloud/the-outbox-pattern/">idempotency</a> in your handlers. Newer sidecar-less / per-node (ambient) designs aim to cut the overhead.
</Callout>
