<Callout type="analogy" title="🪂 Analogy">
Think of `defer` as packing a parachute the moment you board: you set up the cleanup early so it *always* deploys on the way down, however the flight ends. A `panic` is pulling the ejector seat — everything unwinds fast, but each packed parachute (`defer`) still opens. And `recover`, tucked inside a defer, is the one person who can grab the controls and land the plane instead of crashing.
</Callout>

## defer: cleanup that always runs

A `defer` schedules a call for when the surrounding **function returns**, no matter how it returns. Deferred calls run in **LIFO** order (last deferred, first to run), and crucially their **arguments are evaluated at defer time**, not when the call finally fires:

<Playground client:visible title="defer.go" code={`package main

import "fmt"

func main() {
	// LIFO: this prints 3, 2, 1 — reverse of the defer order.
	for i := 1; i <= 3; i++ {
		defer fmt.Println("deferred", i)
	}

	// Arguments are captured NOW, at defer time.
	x := "first"
	defer fmt.Println("snapshot:", x) // captures "first"
	x = "changed"

	fmt.Println("body done, value is", x)
}`} />

The classic use is pairing acquire/release right next to each other: `f, _ := os.Open(name); defer f.Close()`.

## panic unwinds, recover catches

A `panic` stops normal execution and starts **unwinding** the stack — but it runs every pending `defer` on the way up. If a deferred function calls `recover()`, the unwinding **stops** there, `recover` returns the panic value, and the program continues normally from that function's caller.

<Mermaid code={`graph TD
  A["normal call"] --> B["panic(v)"]
  B --> C["unwind: run deferred funcs (LIFO)"]
  C --> D{"a defer calls recover()?"}
  D -->|"yes"| E["panic stops; return value = v"]
  D -->|"no"| F["keep unwinding"]
  F --> G["reach top: program crashes"]`} />

`recover()` only works **inside a deferred function**. Anywhere else it just returns `nil`. The most useful pattern is to recover at a **boundary** — turning a panic into an ordinary returned error so the rest of the program (a server, a worker pool) keeps running:

<Playground client:visible title="recover.go" code={`package main

import (
	"errors"
	"fmt"
)

// safeDivide turns a panic into a normal error at the boundary.
// The named return 'err' lets the deferred recover assign to it.
func safeDivide(a, b int) (result int, err error) {
	defer func() {
		if r := recover(); r != nil {
			err = fmt.Errorf("recovered: %v", r)
		}
	}()
	return a / b, nil // b == 0 triggers a runtime panic
}

func main() {
	if r, err := safeDivide(10, 2); err == nil {
		fmt.Println("10 / 2 =", r)
	}

	// This would panic; recover converts it to an error instead.
	if _, err := safeDivide(1, 0); err != nil {
		fmt.Println("handled:", err)
	}

	// Program survives the panic and keeps going.
	fmt.Println("still running:", errors.Is(nil, nil))
}`} />

## When (not) to panic

Reach for `panic` only when continuing makes no sense:

- **Yes:** an impossible/unreachable state, a corrupt invariant, a required dependency that failed to initialize at startup (`mustConnect`).
- **No:** expected failures — bad user input, a missing file, a network timeout. Those are **values**: return an `error` and let the caller decide.

<Callout type="gotcha" title="⚠️ Don't use panic/recover as exceptions">
Go is deliberately not an exception language. Using `panic`/`recover` for ordinary control flow makes code surprising and hard to follow. Keep panics rare and recover them at a single boundary (a request handler, a goroutine entry point) so one bad request can't take down the whole process. For everything else, return an `error` — see [errors](/fundamentals/errors/).
</Callout>
