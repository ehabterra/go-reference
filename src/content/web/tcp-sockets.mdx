TCP is a phone call: you dial, a circuit opens, and everything you say arrives at the other end in order, exactly once — or the call drops and you both know. UDP (next page) is a postcard: cheap, fire-and-forget, maybe lost. Most of the web rides on the phone call.

## A reliable byte stream

Go's `net` package models a TCP connection as a `net.Conn` — which is just an `io.Reader` **and** an `io.Writer` with an address and deadlines. A server **listens** and **accepts**; a client **dials**:

<Mermaid code={`graph LR
  C["client<br/>net.Dial"] -->|"3-way handshake"| S["server<br/>net.Listen + Accept"]
  C -->|"Write bytes →"| S
  S -->|"← Write bytes"| C
  C -.->|"Close"| S`} />

```go
// server: accept connections, a goroutine each
ln, err := net.Listen("tcp", ":9000")
if err != nil { log.Fatal(err) }
defer ln.Close()
for {
	conn, err := ln.Accept()
	if err != nil { continue }
	go handle(conn) // one goroutine per connection
}

// client: dial and use the Conn as a Reader/Writer
conn, err := net.Dial("tcp", "example.com:9000")
if err != nil { log.Fatal(err) }
defer conn.Close()
fmt.Fprintln(conn, "hello")
```

## The Conn interface, hands-on

Real sockets need a network, but a `net.Conn` is an *interface* — so we can exercise the exact same Read/Write mechanics **in memory** with `net.Pipe()`, which returns two connected ends. This runs anywhere, no network involved:

<Playground client:visible title="conn.go" code={`package main

import (
	"bufio"
	"fmt"
	"io"
	"net"
)

func main() {
	// net.Pipe gives two ends of an in-memory connection — the SAME
	// net.Conn interface a TCP socket implements, with no network at all.
	server, client := net.Pipe()

	// the "server" side: read one line, write a framed reply, then close
	go func() {
		defer server.Close()
		line, _ := bufio.NewReader(server).ReadString('\n')
		fmt.Fprintf(server, "echo: %s", line)
	}()

	// the "client" side: send a request, read until the server closes
	fmt.Fprintf(client, "hello\n")
	reply, _ := io.ReadAll(client)
	fmt.Print(string(reply)) // echo: hello
	client.Close()
}`} />

Notice the `\n` framing: TCP gives you a stream, so the reader uses a delimiter to find where the message ends. `bufio.Scanner` is the usual tool for line-delimited protocols.

## Deadlines, not blocking forever

A network read can hang indefinitely. Set deadlines so a slow or dead peer can't wedge a goroutine:

```go
conn.SetReadDeadline(time.Now().Add(5 * time.Second))
n, err := conn.Read(buf)
// a timeout surfaces as a net.Error with err.Timeout() == true
```

<Callout type="gotcha" title="🔒 Safe-to-run note + the framing trap">
**These playgrounds run in-process** (`net.Pipe`, no sockets, no real hosts) so they're safe to execute anywhere — the go.dev sandbox blocks real network access regardless. When you run the *fenced* examples locally, bind servers to `localhost`/`:port` you control, never expose a debug server on `0.0.0.0` in production, and always `defer conn.Close()`. The classic bug: assuming one `Write` equals one `Read`. It doesn't — always frame (delimiter or length prefix) and set deadlines.
</Callout>
