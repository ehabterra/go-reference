<Callout type="analogy" title="🕴️ Analogy">
Both attacks here are the "confused deputy." Imagine a security guard who'll fetch any file you name from the locked archive — you can't get in, but he can, so you just ask him for the secret files. <strong>SSRF</strong> is that: your server has access to the internal network, so the attacker asks <em>it</em> to fetch the internal thing. <strong>CSRF</strong> is the mirror image — the deputy is the victim's <em>browser</em>, which carries their login cookie, so a malicious page asks the browser to perform actions as the user. Same trick, different deputy.

## SSRF: making your server the attacker

Many apps take a **user-supplied URL** and fetch it server-side — webhooks, "import from URL," avatar fetchers, link previews, PDF/screenshot renderers. **Server-Side Request Forgery** abuses that: the attacker supplies a URL pointing somewhere they couldn't reach themselves, and your server — trusted on the internal network — fetches it for them.

The classic target is the **cloud metadata endpoint** `169.254.169.254`, which can hand out temporary credentials — turning an innocent URL fetcher into cloud-account compromise. Other targets: `localhost`/`127.0.0.1` internal services, private `10.x`/`192.168.x` admin panels, and `file://` schemes.

## See it: an allowlist-style SSRF guard

The reliable defense is to **validate the resolved IP** and reject private/loopback/link-local ranges (an allowlist of *what's permitted* is even stronger). This runs here:

<Playground client:visible title="ssrf_guard.go" code={String.raw`package main

import (
	"fmt"
	"net"
	"net/url"
)

// blocked returns true if an IP is in a range a public fetcher must never reach.
func blocked(ip net.IP) bool {
	return ip.IsLoopback() || ip.IsPrivate() ||
		ip.IsLinkLocalUnicast() || ip.IsUnspecified()
}

// safeHost validates the host of a user-supplied URL. In production you'd
// resolve the name and dial THAT exact IP to avoid DNS rebinding (TOCTOU).
func safeHost(raw string) error {
	u, err := url.Parse(raw)
	if err != nil {
		return fmt.Errorf("bad url: %w", err)
	}
	if u.Scheme != "http" && u.Scheme != "https" {
		return fmt.Errorf("scheme %q not allowed", u.Scheme)
	}
	ip := net.ParseIP(u.Hostname())
	if ip == nil {
		return fmt.Errorf("must resolve+check IP for host %q (see note)", u.Hostname())
	}
	if blocked(ip) {
		return fmt.Errorf("destination %s is in a blocked range", ip)
	}
	return nil
}

func main() {
	for _, raw := range []string{
		"https://93.184.216.34/data",        // public — OK
		"http://169.254.169.254/latest/meta", // cloud metadata — BLOCK
		"http://127.0.0.1:6379",              // internal Redis — BLOCK
		"http://10.0.0.5/admin",              // private net — BLOCK
		"file:///etc/passwd",                 // bad scheme — BLOCK
	} {
		if err := safeHost(raw); err != nil {
			fmt.Printf("REJECT %-38s %v\n", raw, err)
		} else {
			fmt.Printf("ALLOW  %-38s\n", raw)
		}
	}
}`} />

Notice the function insists on checking the **resolved IP**, not the string. Blacklisting the word `localhost` is useless — `127.0.0.1`, `0.0.0.0`, `2130706433` (decimal), `::1`, and DNS names that resolve internally all bypass it.

<Callout type="gotcha" title="⚠️ DNS rebinding and redirects break naive guards">
Validating the hostname and <em>then</em> making a separate request that resolves it again is a TOCTOU bug: an attacker's domain can return a public IP during your check and <code>127.0.0.1</code> at connect time (<strong>DNS rebinding</strong>). Robust SSRF defenses resolve once and dial that exact IP — in Go, a custom <code>net.Dialer</code> with a <code>Control</code> func (or <code>http.Transport.DialContext</code>) that re-checks the IP at the moment of connection — and they also disable or re-validate <strong>redirects</strong> (set <code>http.Client.CheckRedirect</code>), since a 302 to <code>http://169.254.169.254/</code> sidesteps the front-door check.
</Callout>

## CSRF: making the user's browser the attacker

**Cross-Site Request Forgery** is the mirror image. The victim is logged into your site; a malicious page they visit triggers a request to your site, and the browser **automatically attaches the session cookie** — so the request runs *as the user* (transfer funds, change email). The defenses:

- **Anti-CSRF tokens** — an unpredictable, per-session token required on state-changing requests; a cross-site page can't read it (same-origin policy), so it can't forge a valid request.
- **`SameSite` cookies** — `SameSite=Lax` (a good default) or `Strict` stops the browser from sending the cookie on cross-site requests, neutralizing most CSRF.
- Treat **GET as safe/idempotent** — never change state on GET.

<Callout type="go" title="🐹 The Go toolbox: stdlib IP checks + a token middleware">
For SSRF, the standard library gives you everything: <code>net.ParseIP</code> with <code>IsPrivate</code>/<code>IsLoopback</code>/<code>IsLinkLocalUnicast</code>, and a custom <code>http.Transport</code> with a dialer <code>Control</code> func to enforce the IP at connect time. For CSRF, set your session <code>http.Cookie</code> with <code>SameSite: http.SameSiteLaxMode</code> plus <code>HttpOnly</code> and <code>Secure</code>, and add token middleware — <a href="https://github.com/gorilla/csrf" target="_blank" rel="noopener">gorilla/csrf</a> is the well-worn choice. Both attacks are "confused deputy" problems, so the fix is the same shape: don't trust where a request is told to go, and prove the request really came from your own front-end.
</Callout>
