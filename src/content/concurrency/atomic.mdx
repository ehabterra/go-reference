<Callout type="analogy" title="🎟️ Analogy">
A mutex locks the whole room so one person edits the ledger at a time. An atomic counter is a turnstile: each person pushes through and the count ticks up indivisibly — no room to lock, no door to wait at. Great for *one* number; useless if you need several things to change together.
</Callout>

## One value, no lock

`sync/atomic` performs indivisible operations on a single value. The racy counter from [Race Conditions](/concurrency/race-conditions/) becomes correct — and lock-free. Prefer the **typed atomics** (Go 1.19+); they prevent mixing atomic and plain access. **Run** it:

<Playground client:visible title="atomic.go" code={String.raw`package main

import (
	"fmt"
	"sync"
	"sync/atomic"
)

func main() {
	var counter atomic.Int64 // typed atomic — always accessed atomically
	var wg sync.WaitGroup

	for i := 0; i < 1000; i++ {
		wg.Add(1)
		go func() {
			defer wg.Done()
			counter.Add(1) // indivisible, lock-free increment
		}()
	}

	wg.Wait()
	fmt.Println("counter =", counter.Load()) // always 1000
}`} />

## The operations

| Operation | What it does |
|---|---|
| `Load()` | read the current value atomically |
| `Store(v)` | write a value atomically |
| `Add(n)` | atomic increment/decrement (returns the new value) |
| `Swap(v)` | set new, return old, atomically |
| `CompareAndSwap(old, new)` | set new **only if** current == old; returns whether it swapped |

Beyond integers: `atomic.Bool` for flags, `atomic.Pointer[T]` for swapping whole values (config reloads), and the older `atomic.Value` for arbitrary types.

## Compare-and-swap — lock-free updates

CAS is the heart of lock-free algorithms: read, compute, and commit only if nobody changed the value underneath you — otherwise retry.

<Mermaid code={String.raw`graph LR
  R["read old"] --> C{"CAS(old, new)?"}
  C -->|swapped| D["done"]
  C -->|value changed| R`} />

```go
// Atomically take the max, lock-free
for {
	old := best.Load()
	if candidate <= old || best.CompareAndSwap(old, candidate) {
		break
	}
}
```

<Callout type="gotcha" title="⚠️ Atomics guard one variable — not an invariant">
The moment correctness depends on **two** values agreeing (e.g. a slice's length and its backing data, or "balance and last-transaction"), atomics can't help — you need a `Mutex` so the pair changes together. Also: never mix atomic and non-atomic access to the same variable, and on 32-bit platforms misaligned 64-bit fields panic (the typed atomics avoid this for you).
</Callout>

<Callout type="go" title="🐹 Atomic vs mutex — a rule of thumb">
One counter or flag on a hot path → **atomic**. A handful of fields that must stay consistent → **mutex**. Don't reach for atomics to shave nanoseconds off code that isn't hot; measure first with `-race` and `pprof`.
</Callout>
