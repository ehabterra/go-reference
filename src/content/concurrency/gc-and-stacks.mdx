Old garbage collectors shut the whole building to sweep — everyone stops working (a long stop-the-world pause). Go's GC is a cleaning crew that works **alongside** you, only briefly asking everyone to freeze for a moment. And each goroutine's desk (stack) is expandable: it starts tiny and the runtime swaps in a bigger desk the instant you need more room.
</Callout>

## The concurrent garbage collector

Go's GC is a **concurrent, tri-color mark-sweep** collector. The crucial word is *concurrent*: it marks live objects while your goroutines keep running, using a **write barrier** to track pointers that change mid-cycle. Only two short **stop-the-world** phases remain, typically **well under a millisecond** (often tens of microseconds) — the reason Go suits low-latency services.

Two knobs:

- **`GOGC`** (default 100) — run GC when the heap has grown 100% since the last cycle. Higher = less frequent GC, more memory.
- **`GOMEMLIMIT`** (Go 1.19+) — a soft memory ceiling; the GC works harder as you approach it, great for containers.

## Growing stacks

A goroutine starts with a **~2 KB** stack (vs ~1 MB for an OS thread). When it's about to overflow, the runtime allocates a larger stack, **copies** the frames over, and fixes up pointers — and shrinks it back later when it can. That's the secret behind running **millions** of goroutines.

## Escape analysis — stack is free, heap is not

The compiler decides where each value lives. If it can prove a value doesn't outlive its function, it stays on the **stack** — zero allocation, zero GC. If the value "escapes" (returned by pointer, captured by a goroutine, stored in an interface), it goes on the **heap** and the GC manages it.

```bash
go build -gcflags=-m ./...   # prints "escapes to heap" / "does not escape"
```

	"fmt"

func main() {
	// allocate a few MiB of immediate garbage

		_ = make([]byte, 1<<20) // 1 MiB, unreferenced → collectable

<Callout type="go" title="🐹 Reduce allocations before you tune the GC">
The biggest GC win is usually **allocating less**: reuse buffers (a [`sync.Pool`](/concurrency/sync-package/)), preallocate slices with a capacity, and keep values on the stack. Profile with `go test -bench . -benchmem` and `pprof` before reaching for `GOGC`/`GOMEMLIMIT`. Tuning knobs without measuring usually just moves the problem.
</Callout>
