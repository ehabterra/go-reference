<Callout type="analogy" title="🏦 Analogy">
Two tellers update the same account balance from a sticky note. Both read "100", both add 50, both write "150". Two deposits happened, but the balance only went up once — $50 vanished. Nobody enforced "read, then write, before anyone else reads." That's a data race.
</Callout>

## What a data race is

A **data race** happens when two or more goroutines access the same memory concurrently, at least one access is a write, and there's **nothing enforcing an order** between them. The result is undefined — and Go's memory model explicitly says so.

## Atomicity: `i++` is a lie

Statements that *look* indivisible usually aren't. `counter++` is really three operations:

```text
1. read counter        (load)
2. add 1               (modify)
3. write counter back  (store)
```

Two goroutines can both perform step 1 before either reaches step 3 — and one increment is lost.

## Watch it lose updates

A thousand goroutines each increment an **unguarded** counter. It almost never reaches 1000. Edit and **Run** it a few times:

<Playground client:visible title="race.go" code={String.raw`package main

import (
	"fmt"
	"sync"
)

func main() {
	var counter int
	var wg sync.WaitGroup

	for i := 0; i < 1000; i++ {
		wg.Add(1)
		go func() {
			defer wg.Done()
			counter++ // DATA RACE: unsynchronized read-modify-write
		}()
	}

	wg.Wait()
	fmt.Printf("counter = %d (wanted 1000 — the race lost some)\n", counter)
}`} />

## The fixes

Pick one — all three enforce an order:

```go
// 1. Mutex — guard the critical section
var mu sync.Mutex
mu.Lock(); counter++; mu.Unlock()

// 2. sync/atomic — lock-free for a single value
var counter int64
atomic.AddInt64(&counter, 1)

// 3. A channel — let one goroutine own the counter
counts <- 1 // a single owner goroutine does the ++
```

<Callout type="gotcha" title="⚠️ time.Sleep is NOT synchronization">
Sprinkling `time.Sleep` to "let the other goroutine finish" only reshuffles timing — the race is still there and will surface under real load, on a faster machine, or at the worst possible moment. Synchronization means a **mutex, a channel, or `sync/atomic`** — never a sleep.
</Callout>

<Callout type="go" title="🐹 Run everything with -race">
Go ships a world-class **race detector**. Run `go run -race ./...`, and especially `go test -race` in CI. It instruments memory accesses and prints the exact two goroutines, the conflicting addresses, and full stack traces. It only reports races it actually observes at runtime — so exercise your concurrent paths under test.
</Callout>

<Callout type="tip" title="✅ A guarded lock still isn't 'correct'">
Locks fix concurrent *access*, not *logic*. You can have perfectly synchronized code that's still wrong (check-then-act bugs, wrong invariants). Synchronization is necessary, not sufficient — design the logic too.
</Callout>
