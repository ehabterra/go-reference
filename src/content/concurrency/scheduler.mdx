<Callout type="analogy" title="👩‍🍳 Analogy">
A kitchen has a few **stations** (P), a pool of **cooks** (M), and a stream of **orders** (G). A cook works one station, pulling orders from that station's ticket rail. If a cook's rail empties, they grab tickets from a busier station (work-stealing). If a cook has to step away to the walk-in freezer (a blocking syscall), another cook takes over the station so it never sits idle.
</Callout>

## The M:N model

Most languages map one goroutine-equivalent to one OS thread. Go multiplexes **many goroutines onto few threads** — an *M:N* scheduler — with three actors:

- **G — goroutine:** your concurrent function and its tiny stack.
- **M — machine:** an OS thread that actually executes code.
- **P — processor:** a logical CPU that owns a **run queue** of goroutines. To run Go code, an M must hold a P.

You have **`GOMAXPROCS`** Ps (default = number of CPUs). That's the cap on true parallelism.

## Work-stealing keeps cores busy

When a P's run queue empties, it doesn't go idle — it **steals** half the goroutines from another P (or pulls from the global queue). This balances load with no central scheduler bottleneck.

## Blocking and preemption

- **Blocking syscalls:** when a G makes a blocking syscall, its M parks with it, and the P is handed to another M so the remaining goroutines keep running. This is why blocking I/O doesn't freeze your program.
- **Preemption:** early Go could only switch goroutines at function calls, so a tight loop with no calls could hog a P. Since **Go 1.14**, the runtime preempts goroutines **asynchronously** (via signals), so even `for {}` can be interrupted.

## Peek at the runtime

<Playground client:visible title="scheduler.go" code={String.raw`package main

import (
	"fmt"
	"runtime"
	"sync"
)

func main() {
	fmt.Println("NumCPU:    ", runtime.NumCPU())
	fmt.Println("GOMAXPROCS:", runtime.GOMAXPROCS(0)) // 0 = read, don't change

	var wg sync.WaitGroup
	for i := 0; i < 100_000; i++ {
		wg.Add(1)
		go func() { defer wg.Done() }() // 100k goroutines on a few threads
	}
	fmt.Println("live goroutines (incl. main):", runtime.NumGoroutine())
	wg.Wait()
	fmt.Println("done — they all ran on", runtime.GOMAXPROCS(0), "Ps")
}`} />

<Callout type="go" title="🐹 You almost never set GOMAXPROCS — except in containers">
The default (one P per CPU) is right for almost everything; manually pinning it usually hurts. The classic pitfall is **containers**: before **Go 1.25**, the runtime read the host's CPU count, not your cgroup CPU limit — so a 2-CPU-limited pod on a 64-core node spun up 64 Ps and thrashed. **Go 1.25 fixed this by default**: on Linux the runtime now caps `GOMAXPROCS` at the cgroup CPU limit (and never below 2). On older versions, set `GOMAXPROCS` yourself or use `go.uber.org/automaxprocs`.
</Callout>

<Callout type="tip" title="✅ Treat goroutines as cheap, threads as precious">
Goroutines are nearly free (see [Goroutines](/concurrency/goroutines/)). OS threads are not — the runtime grows the M pool when goroutines block in syscalls, and a flood of blocking calls can spawn many threads. Keep blocking work bounded (a [worker pool](/patterns/worker-pool/)) so the thread count stays sane.
</Callout>
