<Callout type="analogy" title="🛰️ Analogy">
A goroutine is a field agent who can't shout back across the building. To report a failure it has to send a message back through a channel — and a good message says *what* failed, *where*, and *whether to retry* — not just "something went wrong."
</Callout>

## The problem: `go f()` swallows the return

Launching `go doWork()` throws away whatever `doWork` returns, error included. You need an explicit path for the error to travel home. Two idioms cover almost everything.

## The Result pattern

Bundle the value and its error in one struct and send it on a channel. The consumer decides what to do with each. **Run** it:

<Playground client:visible title="results.go" code={String.raw`package main

import (
	"errors"
	"fmt"
)

// Result carries either a body or an error — never decide for the caller.
type Result struct {
	URL  string
	Body string
	Err  error
}

func fetchAll(urls []string) <-chan Result {
	out := make(chan Result)
	go func() {
		defer close(out)
		for _, u := range urls {
			if u == "bad" {
				out <- Result{URL: u, Err: errors.New("connection refused")}
				continue
			}
			out <- Result{URL: u, Body: "200 OK"}
		}
	}()
	return out
}

func main() {
	for r := range fetchAll([]string{"a", "bad", "c"}) {
		if r.Err != nil {
			fmt.Printf("%-3s → error: %v\n", r.URL, r.Err)
			continue
		}
		fmt.Printf("%-3s → %s\n", r.URL, r.Body)
	}
}`} />

The key discipline: the goroutine **reports** the error and lets the main goroutine **decide** (retry, skip, abort). It never logs-and-swallows on its own.

## errgroup — wait, first error, cancel the rest

When you fan work out and any task may fail, [`golang.org/x/sync/errgroup`](/patterns/errgroup/) is the tool: `g.Go(func() error {…})`, and `g.Wait()` returns the first error while a shared `context` cancels the siblings.

<Mermaid code={String.raw`graph TD
  G["errgroup.WithContext"] --> A["g.Go: task A"]
  G --> B["g.Go: task B (fails)"]
  G --> C["g.Go: task C"]
  B -. "first error → cancel ctx" .-> A
  B -. cancel ctx .-> C
  G --> W["g.Wait() → returns B's error"]`} />

## What a good error carries

<Callout type="go" title="🐹 Wrap, don't stringify">
Use `fmt.Errorf(\"fetch %s: %w\", url, err)` to **wrap** — the `%w` keeps the cause inspectable via `errors.Is` (sentinel match) and `errors.As` (type match). Need *all* the failures, not just the first? `errors.Join(err1, err2)` (Go 1.20) combines them. A good error tells you **what** happened, **where**, whether it's **retriable**, and a message safe to show a user — separate from the internal detail.
</Callout>

<Callout type="gotcha" title="⚠️ A panic in a goroutine kills the whole program">
`recover` only works in the **same** goroutine that panicked — a parent can't recover a child's panic. If a goroutine runs untrusted or risky code, defer a `recover` *inside* it and turn the panic into an error you send home. Otherwise one goroutine's panic takes down the entire process.
</Callout>
