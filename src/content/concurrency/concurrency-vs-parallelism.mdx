<Callout type="analogy" title="☕ Analogy">
One barista taking an order, then starting the espresso, then steaming milk, then taking the next order is **concurrent** — juggling several tasks, switching between them, one person. Add a second barista working alongside and you have **parallelism** — two tasks happening at the same instant. Concurrency is how you organize the work; parallelism is how many hands do it.
</Callout>

## The core distinction

> **"Concurrency is a property of the code; parallelism is a property of the running program."** — Katherine Cox-Buday, *Concurrency in Go* (O'Reilly, 2017), p. 23

- **Concurrency** is *structure*: composing independent activities that *could* run in any order or overlap.
- **Parallelism** is *execution*: those activities actually running at the same instant on multiple cores.

Concurrent code on a single core is **interleaved**, not parallel — the runtime rapidly switches between tasks. Give it more cores (via `GOMAXPROCS`) and the same concurrent code becomes parallel, with no rewrite.

## CSP — the model Go chose

Most languages stop their concurrency abstraction at **OS threads** and shared memory guarded by locks. Go reaches back to Tony Hoare's 1978 **Communicating Sequential Processes**: model your program as independent processes that **communicate over channels** rather than share memory.

<Callout type="go" title="🐹 The Go proverb">
**"Don't communicate by sharing memory; share memory by communicating."** Go gives you two new primitives below the OS thread — **goroutines** (cheap concurrent functions) and **channels** (typed communication) — so you can model concurrency at the level of your *problem*, not at the level of thread management.
</Callout>

## Channels or mutexes?

Both coordinate goroutines; pick by intent:

| Scenario | Use |
|---|---|
| Transferring **ownership** of data between goroutines | **Channel** |
| Guarding a struct's **internal state** | **Mutex** |
| Coordinating multiple pieces of logic | **Channel** |
| A tight, performance-critical section | **Mutex** (channels synchronize internally) |

## See concurrency in action

Three goroutines run concurrently; the runtime interleaves them, so the order is **not** deterministic. Edit and **Run** (run it twice — the order can change):

<Playground client:visible title="concurrency.go" code={`package main

import (
	"fmt"
	"sync"
)

func main() {
	var wg sync.WaitGroup
	for _, task := range []string{"take order", "pull espresso", "steam milk"} {
		wg.Add(1)
		go func(t string) { // each task is an independent, concurrent activity
			defer wg.Done()
			fmt.Println("doing:", t)
		}(task)
	}
	wg.Wait() // join: wait for all of them
	fmt.Println("all tasks done — order is not guaranteed")
}`} />

<Callout type="gotcha" title="⚠️ More goroutines ≠ faster">
Concurrency is a *design tool*, not automatically a speedup. **Amdahl's Law** says your maximum speedup is capped by the part that must stay sequential, and goroutines add scheduling and communication overhead. Reach for concurrency to model independent work and to overlap I/O — not as a reflex to make everything "parallel."
</Callout>
