<Callout type="analogy" title="🏃 Analogy">
A relay race: the second runner may not start until the baton is in hand. The baton is **synchronization** — passing it is what guarantees "I'm done, and here's the state." Without passing a baton, the next runner has no promise about where the first one got to. In Go, channels and locks *are* the baton.
</Callout>

## What the memory model is

The Go memory model answers one question: **when is a read guaranteed to see a particular write?** The answer is the **happens-before** relation. If event A happens-before event B, then B sees everything A did. If there's *no* happens-before edge between a write in one goroutine and a read in another, you have a [data race](/concurrency/race-conditions/) and the result is undefined — full stop.

## Within one goroutine

Inside a single goroutine, statements happen-before in **program order** (the compiler may reorder, but never in a way you could observe from that goroutine). The subtlety is **across** goroutines — where program order tells you nothing.

## The edges that cross goroutines

These are the operations that establish happens-before *between* goroutines:

| Synchronizing event | The guarantee |
|---|---|
| **Channel send → receive** | a send happens-before the receive that gets that value completes |
| **Channel close → receive** | the close happens-before a receive that returns the zero value |
| **Mutex `Unlock` → next `Lock`** | the unlock happens-before the next lock returns |
| **`sync.Once.Do`** | the one execution happens-before every `Do` returns |
| **`WaitGroup`** | `Done()` calls happen-before `Wait()` returns |
| **`go` statement** | it happens-before the new goroutine starts |
| **`atomic`** | a synchronizing atomic write happens-before the read that observes it |

## Proof: the baton is the channel

Here the close-before-receive edge guarantees the receiver sees the write to `msg`. It is **always** "hello", never the empty string. **Run** it:

<Playground client:visible title="memory_model.go" code={String.raw`package main

import "fmt"

func main() {
	var msg string
	done := make(chan struct{})

	go func() {
		msg = "hello" // (1) write
		close(done)   // (2) close happens-before the receive below
	}()

	<-done           // (3) after this, (1) is guaranteed visible
	fmt.Println(msg) // always "hello" — never ""
}`} />

Delete the channel and read `msg` directly after `go func(){ msg = "hello" }()` and you have a race: it might print `hello`, might print `""`, might be flagged by `-race`. The channel isn't decoration — it's the synchronization that makes the write visible.

<Callout type="gotcha" title="⚠️ 'It worked' is not 'it's correct'">
Racy code can produce the right answer a million times and then fail in production on a different CPU, a busier machine, or after a compiler upgrade. Go has **no `volatile`**, and you cannot reason your way to visibility with `time.Sleep` or clever ordering. The only guarantees are the happens-before edges above.
</Callout>

<Callout type="go" title="🐹 You rarely cite the memory model — you obey it">
In practice you don't reason about happens-before directly; you reach for a [channel](/concurrency/channels/), a [`sync`](/concurrency/sync-package/) primitive, or an [atomic](/concurrency/atomic/) — each of which *is* a happens-before edge — and run `go test -race`. The memory model is the why beneath all of them.
</Callout>
