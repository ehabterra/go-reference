<Callout type="analogy" title="🔑 Analogy">
If channels are conveyor belts moving parts between workers, the `sync` package is the set of **locks on the shared tool cabinet**. When several workers reach into the *same* drawer (a struct's fields), a lock makes sure only one hand is inside at a time. Different job, different tool.
</Callout>

## Mutex & RWMutex — guard internal state

A `sync.Mutex` makes a critical section exclusive. Wrap shared state in a type and lock around access — the same counter that raced [earlier](/concurrency/race-conditions/) is now correct. **Run** it:

<Playground client:visible title="sync.go" code={String.raw`package main

import (
	"fmt"
	"sync"
)

// Counter guards its own state — atomicity scoped to the type.
type Counter struct {
	mu sync.Mutex
	n  int
}

func (c *Counter) Inc() {
	c.mu.Lock()
	defer c.mu.Unlock()
	c.n++
}

func main() {
	c := &Counter{}
	var wg sync.WaitGroup
	for i := 0; i < 1000; i++ {
		wg.Add(1)
		go func() { defer wg.Done(); c.Inc() }()
	}
	wg.Wait()
	fmt.Println("counter =", c.n) // always 1000
}`} />

**`sync.RWMutex`** allows many concurrent readers *or* one writer — a win for read-heavy data (`RLock`/`RUnlock` for reads, `Lock`/`Unlock` for writes). It only pays off when reads vastly outnumber writes.

## The rest of the toolkit

| Type | What it's for |
|---|---|
| **`WaitGroup`** | The join point of fork-join — `Add`, `Done`, `Wait`. |
| **`Once`** | Run something exactly once (lazy init) — the basis of [Singleton](/patterns/singleton/). |
| **`Cond`** | Wait for / signal a condition: `Wait`, `Signal`, `Broadcast`. |
| **`Pool`** | Reuse expensive objects to cut allocations and GC pressure. |
| **`atomic`** | Lock-free reads/writes of a single value (`AddInt64`, `CompareAndSwap`, `atomic.Pointer`). |

```go
// Once — runs the init exactly once, concurrency-safe
var once sync.Once
once.Do(func() { /* expensive setup */ })

// Cond — wake waiters when a condition becomes true
c := sync.NewCond(&sync.Mutex{})
c.L.Lock()
for !ready { c.Wait() } // releases the lock, sleeps, re-acquires on wake
c.L.Unlock()
```

<Callout type="go" title="🐹 The rules that keep sync safe">
**Don't copy** a `Mutex`, `WaitGroup`, `Once`, or `Cond` after use — share by pointer (`go vet` catches copies). **Lock for the shortest time** and never do I/O or sleep while holding a lock (that's how starvation starts). And **don't `Add` to a `WaitGroup` from inside the goroutines it's counting** — call `Add` before the `go` statement.
</Callout>

<Callout type="tip" title="✅ Go's guidance">
Go's own guidance is to prefer channels when they fit the problem, and reach for the `sync` primitives when they're clearly simpler. Mutexes aren't a code smell — for protecting a type's own fields they're often the cleanest, fastest choice. Match the tool to the job, not to dogma.
</Callout>
